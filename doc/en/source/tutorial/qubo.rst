Tensor train optimization for QUBO problems
===========================================

This tutorial describes how to use the TTOpt algorithm in solving quadratic unconstrained binary optimization (QUBO) problems. This example is concerned with the optimization of functions with discrete parameters. For a tutorial describing an application to functions with continuous parameters, please see :doc:`ttopt`.

Preparation
~~~~~~~~~~~

The QUBO problem instances are generated using the Python package ``qubogen``, so the package must be installed to run this tutorial. The package can be installed using the following command:

.. code-block::

    $ python3 -m pip install qubogen

In addition, the packages ``numpy`` (which should already be installed as a prerequisite to ODAT-SE), ``pandas`` and ``networkx`` are required to run this tutorial.

MATLAB is used as a reference solver to compare against, but it is not necessary.

Sample files
~~~~~~~~~~~~

Sample files are available from ``sample/qubo`` .
This directory includes the following files:

- ``run_qubo_test.py``

  Main program file.

- ``qubo_mats.py``

  Auxiliary Python script that dumps the QUBO problem instances (in MATLAB's ``.mat`` format) to a folder.

- ``solve_qubo_mats_tabu.m``

  Auxiliary MATLAB script that solves the QUBO problem instances generated by ``qubo_mats.py`` (and used in ``run_qubo_test.py``) using a MATLAB solver implementing tabu search.

- ``compare_results.py``

  Auxiliary Python script that compares the results of the ODAT-SE run with the results of the MATLAB solver.

- ``qubo_mats_sol.zip``

  ZIP archive containing the output of ``solve_qubo_mats_tabu.m`` in case it is inconvenient to run the MATLAB solver.


Input files
~~~~~~~~~~~

In this example, the input TOML files are dynamically generated for each type of QUBO problem. This subsection describes the input file template.

.. code-block::

    [base]
    dimension = {dim}
    output_dir = "{output_dir}"

    [solver]
    name = "analytical"
    function_name = "{func_name}"

    [algorithm]
    name = "ttopt"
    seed = {12345+j}

    [algorithm.param]
    max_list = {max_list}
    min_list = {min_list}

    [algorithm.ttopt]
    p_points = {p_points}
    q_points = {q_points}
    r_max = 4
    max_f_eval = 100000

Parameter values of the form ``{...}`` are metavariables that are replaced with the appropriate quantity as specified in ``run_qubo_test.py``. For details, see the manual entries :doc:`../input` and :doc:`../algorithm/ttopt`. The variable ``j`` in the expression ``{12345+j}`` refers to an internal counter over trials, so that for a given problem instance, multiple trials with different starting PRNG seeds can be considered.

The ``[base]`` section specifies some global parameter values used in the ODAT-SE run:

- ``dimension`` denotes the number of dimensions in the optimization. This corresponds to the number of scalar parameters needed to specify a point in the state space.

- ``output_dir`` is the location where the results of the ODAT-SE run will be saved to.

The ``[algorithm]`` section specifies general parameters for the algorithm.

- ``name`` is the name of the algorithm (``ttopt`` in this case)

- ``seed`` is the initial seed used in the PRNG.

The ``[algorithm.param]`` section sets the parameter space to be explored.

- ``max_list`` is a list of upper bounds along each dimension in the optimization.

- ``min_list`` is a list of lower bounds along each dimension in the optimization.

The ``[algorithm.ttopt]`` section specifies parameters specific to the TTOpt algorithm.

- ``p_points`` is the number of modes for each dimension, which corresponds to the bond dimension along the tensor legs allotted for the parameter. Together with the ``q_points`` parameter, this is related to the number of possible values each parameter can take. The total number of possible values for a parameter is :math:`p^q`.

- ``q_points`` is the number of submodes for each dimension, which corresponds to the number of tensor legs allotted for the parameter. Together with the ``p_points`` parameter, this is related to the number of possible values each parameter can take. The total number of possible values for a parameter is :math:`p^q`.

- ``r_max`` is the maximum rank of the implicit MPS representation used to model the state space.

- ``max_f_eval`` is the maximum number of function evaluations used in the optimization process.


Calculation
~~~~~~~~~~~

We assume that the current working directory is the directory containing the relevant tutorial files.

.. code-block::

   $ cd sample/qubo
   
This tutorial is concerned with finding candidate minima for certain types of QUBO problems in :math:`d=50,100,200` dimensions. Specifically, we focus on instances describing the max-cut, minimum vertex cover, and quadratic knapsack problems. In the language of spin systems, these QUBO-type problems correspond to (possibly frustrated) systems of :math:`d` Ising spins. The test program is run by executing the following command together with an optional argument that sets the number of MPI processes to use:

.. code-block::

   $ export OMP_NUM_THREADS=1; mpiexec -np 8 python3 ./run_qubo_test.py | tee log.txt

Here, the optimization is performed using 8 processes communicating over MPI. The optimization may take some time to run, particularly for the larger problem sizes.

For each benchmark function, a folder of the form ``output/output_{func_name}_{dim}_{instance}`` is first created, and all output files for the ODAT-SE run are stored in this folder. Each MPI process is given a subfolder indexed by the process rank. Within each subfolder is a log file containing execution time details for the process. The log file containing the optimization history for the entire optimization process is the ``ttopt_history.txt`` file located in the output folder.

The file ``ttopt_history.txt`` contains a list of parameters set when invoking the optimization method, as well as a record of the number of function evaluations, the best point found so far, and the best objective function value found so far.

Running the main Python script also generates two CSV files, ``qubo_results.csv`` and ``qubo_results_agg.csv``. The former contains the result of each trial for all instances of a problem at a given dimension, and the latter contains aggregated results for the minimum function value obtained for each problem and dimension.

To examine the quality of the obtained solutions, we consider an alternate solution using an established solver like MATLAB's tabu search-based solver. The next two commands can be skipped if relying on the results found in ``qubo_mats_sol.zip``, in which case the contents of the archive can be decompressed into the working directory. Note that the MATLAB solver is a stochastic solver, so its output may differ slightly each time it is run.

To obtain the QUBO instance matrices used in ``run_qubo_test.py``, we run the following command:

.. code-block::

   $ python3 ./qubo_mats.py

Next, the QUBO instance matrices are passed to the MATLAB solver, which can be run in the MATLAB interactive terminal or as a script. The MATLAB solver can be run in a shell using the following command:

.. code-block::

   $ matlab -nodisplay -nosplash -nodesktop -r "run('solve_qubo_mats_tabu.m');"

We then run the auxiliary script ``compare_results.py`` to obtain a comparison of the results from both approaches:

.. code-block::

   $ python3 ./compare_results.py

The script computes the relative difference (using the MATLAB solver result as the reference) in the estimated minimum objective value for each instance, and the relative difference for the aggregated results.
